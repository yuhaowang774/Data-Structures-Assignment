### 1.2 核心内容熟悉

**必须烂熟于心的三个问题：**

#### 问题1：你的优先队列（最小堆）是怎么实现的？insert 和 extract-min 的时间复杂度是多少？

**回答要点：**

- 最小堆用一维数组存储，下标从0开始
- 节点 i 的父节点为 `floor((i-1)/2)`，左子节点为 `2i+1`，右子节点为 `2i+2`
- **insert 操作**：新元素插入数组末尾，然后上浮（swim）——与父节点比较，若更小则交换，直到堆性质恢复。最坏情况从叶节点上浮到根，路径长度为堆高度 log n，**时间复杂度 O(log n)**
- **extractMin 操作**：取出堆顶（最小值），将数组末尾元素移到堆顶，然后下沉（sink）——与两个子节点中较小者比较，若比子节点大则交换，直到堆性质恢复。同样路径长度最多 log n，**时间复杂度 O(log n)**
- 堆中存储的是状态节点 `{nodeId, cost}`，cost 是 Dijkstra 的优先级值

**代码对应（script.js L85-137）：**

```
insert: push → 上浮循环（L89-100）
extractMin: 取堆顶 → 末尾补顶 → 下沉循环（L102-133）
isEmpty: 判空（L135-137）
```

#### 问题2：为什么不能直接用 BFS 找最优路径？BFS 能找到换乘最少的路径吗？

**回答要点：**

- **BFS 只能处理等权图**（所有边权重相同），每次扩展一层，先到达终点的就是最短路径
- 地铁网络中**边权不等**：
  - 相邻站行驶时间不同（1.5-3分钟不等）
  - 换乘边权重为5分钟（含2分钟换乘等待）
  - 高峰期还有时间系数调整
- 用 BFS 会把所有边视为等权，**无法区分2分钟和5分钟的边**，导致找到的路径可能不是时间最短的
- **BFS 能否找换乘最少？** 理论上如果把换乘边权重设为1、非换乘边权重设为0，BFS 可以找换乘最少，但这需要改造图结构，且无法同时考虑时间因素
- **扩展 Dijkstra 的优势**：通过 cost 函数设计，可以灵活支持三种优化目标——时间最短（mode=0）、换乘最少（mode=1）、综合最优（mode=2），且正确处理不等权边

### 1.3 Dijkstra 算法执行过程（超详细版）

> 如果你完全不懂算法，按这个顺序读：先看"生活类比"，再看"核心思路"，最后看"逐步执行"。

#### 第一步：用生活类比理解 Dijkstra

想象你要从家去一个陌生的地方，城市里有很多条路，每条路走的时间不同。你不知道哪条路最快，怎么办？

**笨办法**：把所有可能的路线都走一遍，比较总时间。路线多了根本算不过来。

**Dijkstra 的聪明办法**：从家出发，每次都先探索"当前已知最近"的那个地方。因为如果连最近的地方都没走到，更远的地方就更不可能是最优路线了。

核心思想就一句话：**永远先处理当前最近的未访问节点，它的最短距离已经确定了，不可能再更短。**

#### 第二步：算法需要的"工具箱"

| 工具                      | 作用                   | 类比                  |
| ----------------------- | -------------------- | ------------------- |
| **邻接表 adjList**         | 记录每个站点能到达哪些邻居，以及距离   | 地铁线路图               |
| **距离数组 dist\[]**        | 记录从起点到每个站点的最短时间      | 每个站贴个便签，写"目前已知最短时间" |
| **换乘数组 transferArr\[]** | 记录到每个站点已经换乘了几次       | 便签上再写"换乘了几次"        |
| **访问标记 visited\[]**     | 记录哪些站点已经确定了最短路径      | 便签上的值不会再改了，画个✓      |
| **前驱数组 prev\[]**        | 记录每个站点是从哪个站点来的       | 每个站留个箭头，指向上一个站      |
| **最小堆 MinHeap**         | 每次快速取出"当前距离最短"的未访问站点 | 一个排队系统，距离短的排前面      |

#### 第三步：核心思路（伪代码）

```
1. 把起点的距离设为 0，其他所有站点距离设为 ∞（无穷大）
2. 把起点放入最小堆
3. 循环：
   a. 从最小堆中取出距离最小的站点 → 叫它"当前站"
   b. 如果当前站已经访问过，跳过
   c. 标记当前站为已访问
   d. 如果当前站就是终点，结束！
   e. 看看当前站的所有邻居：
      - 计算经过当前站到邻居的新距离 = dist[当前站] + 这条边的时间
      - 计算经过当前站到邻居的新换乘数 = transferArr[当前站] + 这条边是否换乘
      - 计算新的 cost 值（根据模式不同公式不同）
      - 如果新 cost 比邻居原来的 cost 更小：
        · 更新邻居的距离、换乘数、前驱
        · 把邻居放入最小堆
4. 从终点沿着 prev[] 一步步回溯到起点，就是完整路径
```

#### 第五步：三种模式的 cost 公式详解

**为什么需要不同的 cost 公式？** 因为"最优"的定义不同：

**mode=0 时间最短：`cost = totalTime`**

- 只看时间，谁快选谁
- 举例：路径A耗时30分钟换乘3次，路径B耗时31分钟换乘0次 → 选A

**mode=1 换乘最少：`cost = transfers + totalTime × 0.000001`**

- 主要看换乘次数，时间几乎不影响
- transfers 是整数（0,1,2...），totalTime×0.000001 是极小的数
- 举例：路径A换乘2次耗时25分钟(cost≈2.000025)，路径B换乘3次耗时20分钟(cost≈3.000020) → 选A
- 那个 ×0.000001 的作用：换乘次数相同时，用时间来打破平局

**mode=2 综合最优：`cost = totalTime × 1000 + transfers`**

- 主要看时间，时间相同时换乘少的优先
- totalTime×1000 让时间成为主导因素（差0.001分钟就是差1）
- transfers 作为低位，只在时间完全相同时起作用
- 举例：路径A耗时30分钟换乘3次(cost=30003)，路径B耗时30分钟换乘1次(cost=30001) → 选B
- 举例：路径A耗时29分钟换乘3次(cost=29003)，路径B耗时30分钟换乘1次(cost=30001) → 选A（时间差1分钟 > 换乘差2次）

**一张图理解三种模式：**

```
模式0（时间最短）：  cost = 30          → 只比时间
模式1（换乘最少）：  cost = 2 + 0.00003 → 只比换乘
模式2（综合最优）：  cost = 30000 + 2   → 先比时间，时间同比换乘
```

#### 第七步：复杂度分析（答辩必问）

- **时间复杂度**：O((V + E) log V)
  - 每个节点最多入堆一次，每次 insert 是 O(log V) → 总共 O(V log V)
  - 每条边最多被检查一次，可能触发 insert → O(E log V)
  - 合计 O((V + E) log V)
  - 在我们的项目中 V=289, E≈600，所以实际运算量大约 (289+600)×9 ≈ 8000 次堆操作
- **空间复杂度**：O(V + E)
  - 邻接表存所有边：O(V + E)
  - dist/visited/prev 数组：O(V)
  - 堆最多存 V 个元素：O(V)

## 三、问答环节应对策略

### 3.1 常见问题及参考回答

#### Q: 你的邻接表是怎么构建的？

A: 从 graph.json 读取节点和边，节点数组中每个元素包含 id、站名、线路名、经纬度。边数组中每条边记录 from、to、weight（行驶时间）、line（所属线路）、is\_transfer（是否换乘边）。构建时先为每个节点初始化空邻接链表，再遍历边数组将每条边加入对应节点的链表。换乘站会出现在多个节点中（同一站名不同线路各一个节点），换乘边连接同一站名不同线路的节点，权重为5分钟。

#### Q: 换乘等待时间怎么处理的？

A: 换乘边（is\_transfer=1）的权重设为5分钟，包含2分钟换乘等待时间。在高峰/平峰模式下，换乘时间还会乘以对应系数（高峰1.5倍、平峰1.0倍），通过 ScheduleConfig 模块动态调整邻接表中的边权。

#### Q: 综合最优模式和时间最短模式有什么区别？

A: 时间最短模式（mode=0）只优化总时间，cost = totalTime。综合最优模式（mode=2）以时间为主、换乘为辅，cost = totalTime × 1000 + transfers。乘以1000确保时间差0.001分钟就能区分路径，而 transfers 作为低位使得时间完全相同时换乘少的路径优先。这满足了题目"时间相同时优先换乘少"的要求。

#### Q: 你的图有多少节点？为什么不是60多个？

A: 图中有289个节点，因为换乘站按"站名+线路"拆分为多个节点。例如小寨站同时属于2号线和3号线，在图中是两个独立节点，通过换乘边连接。这样设计使得 Dijkstra 扩展时能追踪当前所在线路，正确计算换乘次数和换乘等待时间。实际物理站点约140个，覆盖13条线路。

#### Q: 备用方案是怎么生成的？

A: 使用惩罚避让策略。先求最优路径，然后对路径中的中间站点施加高惩罚（+10000），迫使算法绕开这些站点寻找替代路径。去重时比较路径相似度（>80%相似视为重复），最终按时间+换乘排序展示。

#### Q: 如果两个站点之间没有路径怎么办？

A: Dijkstra 遍历完所有可达节点后 foundNode 仍为 -1，函数返回 `{path: [], error: "No path found"}`，前端显示"未找到路径"提示。
