## 一、图数据模型：graph.json 的结构

**在讲算法之前，必须先搞懂数据是怎么组织的。graph.json 是整个系统的数据基础。**

graph.json 包含两个核心数组：nodes（节点）和 edges（边）。

### 1.1 nodes 数组 —— 图的节点

每个节点代表"某个站名的某条线路"，不是物理站点。换乘站会拆成多个节点。

```json
{
  "id": 131,           // 节点唯一编号，0~288
  "station": "小寨",    // 站名（同一站名可出现多次）
  "line": "3号线",      // 该节点所属线路
  "lat": 34.2271,      // 纬度（WGS84原始坐标）
  "lon": 108.9487      // 经度
}
```

> 🔵 小寨站同时属于2号线和3号线，所以有两个节点：id=95(2号线) 和 id=131(3号线)
> 🔵 它们之间通过一条换乘边连接，权重5分钟
> 🔵 这样设计是为了让 Dijkstra 能追踪"当前在哪条线上"，从而正确计算换乘

### 1.2 edges 数组 —— 图的边

```json
{
  "from": 131,          // 起点节点id
  "to": 132,            // 终点节点id
  "weight": 2.5,        // 边的权重（分钟）
  "line": "3号线",       // 所属线路
  "is_transfer": 0      // 0=行驶边，1=换乘边
}
```

> 🔵 行驶边：相邻两站之间的乘车时间，通常1.5~3分钟
> 🔵 换乘边：同一站名不同线路节点之间的连接，权重5分钟（含2分钟等待）
> 🔵 is_transfer 是关键标志，Dijkstra 用它来计算换乘次数

**🟠 C语言对照：如果用C写，nodes和edges对应的结构体如下**

```c
// C 语言中，nodes 数组的每个元素是这样的结构体

    int id;           // 节点唯一编号
    char station[32]; // 站名
    char line[16];    // 所属线路
    double lat;       // 纬度
    double lon;       // 经度
} Node;

// edges 数组的每个元素
typedef struct {
    int from;         // 起点节点id
    int to;           // 终点节点id
    double weight;    // 边的权重（分钟）
    char line[16];    // 所属线路
    int is_transfer;  // 0=行驶边，1=换乘边

// JS 的 JSON 对象 ≈ C 的结构体实例
// JS: { id: 131, station: "小寨", line: "3号线" }
// C: Node n = { 131, "小寨", "3号线", 34.2271, 108.9487 };
// 区别：C 需要提前定义 struct 类型，JS 随用随写
```

**答辩话术：**

```
"我们的图有289个节点、636条边。为什么不是250个节点？
因为换乘站按'站名+线路'拆分，比如小寨站拆成2号线节点和3号线节点，
中间用换乘边连接。这样Dijkstra扩展时能知道当前在哪条线上，
才能正确计算换乘次数和换乘等待时间。"
```

## 三、MinHeap —— 最小堆（优先队列）

**最小堆是 Dijkstra 高效运行的关键数据结构，用来快速取出"当前距离最小"的节点。**

### 3.1 数据结构：一维数组存储完全二叉树

```
堆的存储：this.heap = []

下标关系（从0开始）：
  节点 i 的父节点 = floor((i-1)/2)
  节点 i 的左子节点 = 2*i + 1
  节点 i 的右子节点 = 2*i + 2

举例：堆中有5个元素 [10, 20, 30, 40, 50]
  下标0(10) 的左子=1(20)，右子=2(30)
  下标1(20) 的左子=3(40)，右子=4(50)
  下标3(40) 的父=floor((3-1)/2)=1(20)

堆性质：每个父节点的cost ≤ 子节点的cost
  → heap[0] 永远是最小值
```

**🟠 C语言对照：MinHeap 的 C 实现方式**

```c
// C 中用结构体 + 函数指针（或独立函数+传指针）
typedef struct {
    int nodeId;
    double cost;
    double totalTime;
    int transfers;
} HeapNode;

typedef struct {
    HeapNode heap[289];
    int size;
} MinHeap;

// JS: function MinHeap() { this.heap = []; }
// C: 用 typedef struct 定义，JS 用构造函数
// JS: this.heap  →  C: h->heap（通过指针访问）
// JS: this.heap.push(node)  →  C: h->heap[h->size++] = node;
// JS: this.heap.pop()       →  C: h->heap[--h->size];
// JS: this.heap.length      →  C: h->size
```

```
> 🔵 记录找到的终点节点id，-1表示还没找到

### 4.6 扩展邻居 —— 松弛操作

```js
    var edges = adjList[cur.nodeId];
    if (!edges) continue;
    for (var j = 0; j < edges.length; j++) {
      var edge = edges[j];
      var neighbor = edge.to;
      if (visited[neighbor]) continue;
```
> 🔵 取出当前节点的所有邻居边，跳过已访问的邻居

```js

```
> ★ **核心：新距离 = 到当前节点的距离 + 边的权重**。这就是"松弛"——尝试经过当前节点到邻居是否更短

```js

```
> 🔵 新换乘数 = 到当前节点的换乘数 + 这条边是否换乘。is_transfer=0时不变，=1时换乘数+1

### 4.7 cost 计算 —— 三种模式的核心区别

```js
      var newCost;
      if (mode === 0) {
        newCost = newTime;                              // 模式0：cost = 纯时间，谁快选谁
      } else if (mode === 1) {
        newCost = newTransfers + newTime * 1e-6;        // 模式1：换乘主导，时间极小值打破平局
      } else {
        newCost = newTime * 1000 + newTransfers;        // 模式2：时间主导，时间相同时换乘少优先
      }
```

> 🔵 mode=1中，transfers是整数(0,1,2...)，totalTime×0.000001极小，所以换乘次数主导
> 🔵 mode=2中，totalTime×1000让时间成为主导因素，transfers作为低位

        : mode === 1 ? transferArr[neighbor] + dist[neighbor] * 1e-6
        : dist[neighbor] * 1000 + transferArr[neighbor])) {
```
> 🔵 比较新cost和旧cost。如果邻居还没被更新过(dist=INF)，新cost一定更小
> 🔵 这里用嵌套三元运算符根据模式计算旧cost，和4.7节的newCost公式对称

```js

        transferArr[neighbor] = newTransfers;
        prev[neighbor] = cur.nodeId;
```
> ★ **关键：记录前驱**，用于最后回溯路径

```js

## 五、dijkstraWithPenalty —— 带惩罚的Dijkstra（备用方案）

**用于生成备用方案。原理：对最优路径上的中间站点施加高惩罚，迫使算法绕开这些站点。**

### 5.1 与普通dijkstra的区别

dijkstraWithPenalty 在 dijkstra 基础上增加了：

1. **realDist[] 数组** —— 记录真实时间（不含惩罚）
2. **penalty 参数** —— 每个节点可以有一个惩罚值
3. 计算newCost时，把惩罚值加到时间上

### 5.2 关键代码差异

```js
  var penalty = (penaltyWeights && penaltyWeights[cur.nodeId]) || 0;
```
> 🔵 取出当前节点的惩罚值，没有则为0
> 🟠 C对照：`double penalty = (penaltyWeights != NULL) ? penaltyWeights[cur.nodeId] : 0;`
> 🟠 JS 的 `&&` 和 `||` 短路求值 ≈ C 的三元运算符，但更简洁

```js
  var newRealTime = realDist[cur.nodeId] + edge.weight;
```
> 🔵 真实时间 = 不含惩罚的纯行驶时间

```js
  var newPenalizedTime = dist[cur.nodeId] + edge.weight + penalty;
```
> 🔵 惩罚时间 = 上一节点的惩罚时间 + 边权 + 当前节点惩罚
> 🔵 penalty让经过被惩罚节点的路径cost变大，算法会尽量绕开

```js
  newCost = newPenalizedTime;  // mode=0时
```
> 🔵 cost用的是含惩罚的时间，但返回的total_time是真实时间

### 5.3 备用方案生成流程（findAlternativePaths）

```
1. 先用 dijkstraWithPenalty(start, end, mode, null) 求最优路径
2. 提取最优路径上的中间站点
3. 对每个中间站点，设置 penalty=10000，重新求路径
4. 去重：如果新路径和已有路径相似度>80%，视为重复，丢弃
5. 按时间+换乘排序，最多返回5个方案
```

> 🔵 penalty=10000 相当于给该站点加了10000分钟惩罚，算法几乎不会经过它

## 六、ScheduleConfig —— 时段调度如何影响算法

**ScheduleConfig 模块在 buildAdjList 阶段就介入了，它修改的是邻接表中的边权。**

### 6.1 工作流程

```
1. 检测当前时段（早高峰/晚高峰/平峰）
   → 根据当前时间自动判断，或手动选择

2. 获取对应系数
   → 高峰：运行×1.3，换乘×1.5
   → 平峰：运行×1.0，换乘×1.0

3. 调整每条边的权重
   → 行驶边：weight × 运行系数
   → 换乘边：weight × 换乘系数

4. 用调整后的邻接表运行 Dijkstra
   → 高峰期换乘5分钟变7.5分钟，算法会更倾向于少换乘
```

### 6.2 关键代码

```js
  function adjustWeight(baseWeight, lineName, isTransfer) {
    var factor = getMultiplier(lineName, isTransfer);        // 获取系数（根据时段和是否换乘）
    return Math.round(baseWeight * factor * 100) / 100;     // 新权重 = 原始权重 × 系数，保留两位小数
  }
```

> 🟠 C对照：
> ```c
> double adjustWeight(double baseWeight, char* lineName, int isTransfer) {
>     double factor = getMultiplier(lineName, isTransfer);
>     return round(baseWeight * factor * 100) / 100;  // C 的 round() ≈ JS 的 Math.round()
> }
> ```
> 🟠 C 要声明返回值类型 double 和参数类型，JS 不需要。Math.round() ≈ C 的 round()

**答辩话术：**

```
"时段调度不是在Dijkstra内部处理的，而是在构建邻接表时就把权重调整好了。
高峰期换乘系数1.5，所以换乘边从5分钟变成7.5分钟，
Dijkstra自然会倾向于少换乘的路径。这是在数据层面的预处理，算法本身不需要改动。"
```

## 七、答辩速查：被问到某段代码怎么说

### 7.1 "讲一下你的MinHeap实现"

```
回答模板：
"最小堆用一维数组存储，下标从0开始。
节点i的父节点是floor((i-1)/2)，左子是2i+1，右子是2i+2。
insert操作：新元素放末尾，和父节点比较，更小就交换，一直上浮，O(log n)。
extractMin操作：取堆顶，末尾补顶，和较小子节点比较，更大就交换，一直下沉，O(log n)。
堆中存的是{nodeId, cost}，cost是Dijkstra的优先级值。"
```

> 🟠 追问"和C语言实现有什么区别"时的补充：
> "逻辑完全一样。C用结构体+函数传指针：`MinHeap_insert(&h, node)`，
> JS用构造函数+原型方法：`h.insert(node)`，this自动绑定等价于C的&h。
> C的整数除法自动取整，JS需要Math.floor()。C手动维护size，JS用push/pop自动管理。"

### 7.2 "讲一下你的Dijkstra实现"

```
回答模板：
"首先从graph.json构建邻接表，每个节点的邻居存在数组里。
然后初始化dist数组为无穷大、visited为false、prev为-1。
把起点入堆，然后循环：从堆中取cost最小的节点，
如果已访问就跳过，否则标记已访问，检查是否到达终点。
没到终点就遍历邻居，计算新的cost（根据模式不同公式不同），
如果比原来的小就更新dist、transferArr、prev，并把邻居入堆。
最后从终点沿prev回溯到起点，得到完整路径。"
```

> 🟠 追问"和C语言实现有什么区别"时的补充：
> "算法逻辑完全一样。主要语法差异：JS用var声明变量不需要类型，
> C要写double dist[]和int visited[]；JS用this代替C的指针传参；
> JS的对象字面量`{nodeId, cost}`等价于C的结构体实例化`(HeapNode){nodeId, cost}`。
> 算法思想、数据结构、时间复杂度在两种语言中完全相同。"

### 7.3 "换乘是怎么处理的？"

```
回答模板：
